# 用户视角验收清单与体验优化建议

## 1. 文档目的

本文件用于两个场景：

- 作为毕业设计阶段的用户验收测试清单，验证系统是否具备完整闭环。
- 作为上线前的产品优化清单，识别当前系统从“可演示”走向“可上线”还缺哪些关键细节。

本文档不只列功能点，而是站在真实用户使用过程评估系统是否顺手、可信、连续、可恢复。

## 2. 验收对象

本次验收面向当前已经实现的系统能力：

- 注册与登录
- 个人档案与健康约束维护
- 首页推荐与推荐理由展示
- 菜谱浏览
- 饮食记录与统计趋势
- 健康报表导出
- 社区发帖、评论、举报

## 3. 验收前置条件

在执行用户验收前，应满足以下环境前提：

- 前端可正常访问登录页与各业务页面。
- 后端 API 服务已启动。
- PostgreSQL 与 Redis 服务已连通。
- 至少存在一批可展示的菜谱基础数据。
- 报表导出目录可写。
- 外部 API 即使未配置真实 key，也能返回降级结构而不是报错。

## 4. 核心用户旅程验收

### 4.1 旅程一：新用户首次进入系统

目标：验证用户能否在第一次使用时快速理解系统价值并顺利开始。

操作步骤：

1. 进入登录页。
2. 输入用户名、邮箱、手机号和密码中的必要信息。
3. 点击“注册并登录”。
4. 进入首页后再进入“个人中心”完善资料。
5. 返回首页查看 BMI、热量目标、蛋白目标和推荐结果。

预期结果：

- 用户可以成功完成注册并自动登录。
- 系统不会在首次进入时出现空白页或无引导状态。
- 首页能看到基础指标和推荐内容。
- 用户能理解系统的主要价值：推荐、记录、分析。

当前观察：

- 登录页文案清晰，注册入口直接可见，适合首次体验。
- 但登录页同时展示“账号、邮箱、手机号”三个输入框，容易让用户误以为登录必须填多个字段。
- 当前登录逻辑实际只使用“账号”输入框，不会读取单独的邮箱和手机号输入框。这会造成真实用户困惑。

结论：

- 该旅程基本可走通。
- 上线前必须优化登录表单，只保留一个统一的“账号”输入框，或明确标注“邮箱和手机号仅用于注册补充信息”。

### 4.2 旅程二：用户完成建档并获得个性化推荐

目标：验证系统能否把个人资料与推荐结果建立联系。

操作步骤：

1. 登录后进入“个人中心”。
2. 填写身高、体重、目标体重、饮食偏好、饮食类型。
3. 勾选糖尿病、高血压、高血脂等健康约束。
4. 填写过敏标签和忌口标签。
5. 点击“保存全部”。
6. 返回首页查看推荐结果和目标提示。
7. 点击某条推荐查看推荐理由。

预期结果：

- 档案保存成功。
- 首页能展示 BMI、热量目标、蛋白目标。
- 不同健康约束下，首页提示语发生变化。
- 推荐理由能够让用户理解系统为什么推荐该菜谱。

当前观察：

- 当前全档案保存接口是原子化的，符合真实业务要求。
- 首页已根据健康约束展示不同提示语。
- 推荐理由通过点击推荐卡片弹出消息提示，基础链路可用。
- 但推荐解释仅以短消息形式展示，缺少详情页、可对比说明和长期反馈入口。

结论：

- 该旅程已具备展示和演示价值。
- 若面向正式产品，推荐解释应沉淀为卡片详情或抽屉面板，而不是只依赖一次性消息弹窗。

### 4.3 旅程三：用户快速记录一餐并查看统计

目标：验证系统是否真正适合日常连续使用。

操作步骤：

1. 进入“记录”页面。
2. 选择日期、餐次和录入方式。
3. 关联一个已有菜谱，或仅填写备注。
4. 点击“保存记录”。
5. 查看顶部营养统计和底部趋势明细。
6. 切换“最近 7 天”和“最近 30 天”。

预期结果：

- 记录应保存成功。
- 新记录应立即出现在列表中。
- 统计与趋势应同步更新。
- 用户能直观看出自己吃得是否合理。

当前观察：

- 记录链路可以跑通，符合 MVP 标准。
- 提供了按餐次、日期、菜谱录入的基本能力。
- 当没有选择菜谱时，系统会把备注内容作为手动条目回填，确保记录能落库。
- 但当前记录页缺少编辑、删除、复制上一餐、真正的扫码录入和条目级营养编辑。
- “快速添加”“复制昨天”“扫码”作为录入方式已出现在界面中，但当前未形成完整差异化交互，容易让用户误以为已经支持完整功能。

结论：

- 该旅程具备基础可用性。
- 上线前需要把未完全落地的入口处理清楚：要么继续实现，要么临时隐藏，避免“点得到但不好用”的伪功能。

### 4.4 旅程四：用户导出阶段性报表

目标：验证系统是否能形成复盘闭环。

操作步骤：

1. 进入“报表”页面。
2. 点击“生成周报”。
3. 查看最近结果中的任务编号、起止日期。
4. 下载生成的最新报表。
5. 再次点击“生成月报”。

预期结果：

- 周报、月报都能生成。
- 下载链接可访问。
- 报表内容与饮食记录、目标进度保持一致。

当前观察：

- 当前页面可以生成周报和月报，并展示任务信息。
- 下载入口明确，满足毕业设计成果展示需要。
- 但生成过程缺少“处理中”状态、失败重试说明和历史报表列表。
- 若前后端部署域名或静态资源路径配置不一致，下载链接在真实线上环境中容易出现跨域或地址错误问题。

结论：

- 当前适合演示和基础验收。
- 上线前建议把报表任务改为真正异步轮询，并加入历史记录与下载失败提示。

### 4.5 旅程五：用户在社区发布和互动

目标：验证社区模块是否具备最基本的活跃与治理能力。

操作步骤：

1. 进入“社区”页面。
2. 发布一条帖子。
3. 对帖子评论。
4. 对帖子发起举报。
5. 刷新列表确认帖子和评论展示。

预期结果：

- 发帖、评论、举报均成功。
- 评论能出现在对应帖子下。
- 举报不影响当前主流程稳定性。

当前观察：

- 社区主链路已具备。
- 刷新加载简单直接，演示效果明确。
- 但目前缺少点赞、收藏、我的帖子、分页、排序、敏感词拦截和审核状态提示。
- 举报后用户看不到处理进度，也不知道平台是否已受理。

结论：

- 当前适合作为“模块能力已实现”的答辩展示。
- 若面向长期运营，社区治理链路还需要后台管理和审核反馈补齐。

## 5. 分模块验收清单

### 5.1 账号模块

- 注册成功后能自动登录。
- 支持使用用户名、邮箱或手机号登录。
- token 丢失后受保护页面会跳转登录页。
- 退出登录后本地 token 被清除。
- 异常密码提示明确。

当前风险：

- 登录页视觉上容易造成多字段误解。
- 尚未实现找回密码、验证码登录、隐私设置。

### 5.2 个人中心模块

- 查看当前账号信息。
- 修改昵称、签名、邮箱、手机号。
- 修改身高、体重、目标体重。
- 配置过敏、忌口和慢病信息。
- 保存后重新进入页面仍能看到已保存数据。

当前风险：

- 字段虽完整，但缺少更细的表单校验提示。
- 当前输入形式偏工程化，缺少枚举型选项控件和填写引导。

### 5.3 首页推荐模块

- 可以拉取推荐列表。
- 可以点击查看推荐理由。
- 可以展示 BMI、热量目标、蛋白目标。
- 当资料不完整时，页面仍能稳定展示。

当前风险：

- 推荐为空时缺少明确的空状态引导。
- 推荐解释承载方式偏轻，不利于建立信任。

### 5.4 菜谱模块

- 菜谱列表可展示标题、描述、餐次、难度、时长。
- 页面可刷新。

当前风险：

- 当前前端未实现搜索、筛选、详情、收藏按钮。
- 从用户视角看，菜谱库更像“浏览页”，还不是“可决策页”。

### 5.5 饮食记录模块

- 可新增记录。
- 可查看统计汇总。
- 可查看趋势数据。
- 可按周期切换统计。

当前风险：

- 缺少编辑与删除。
- 缺少空数据引导和异常恢复说明。
- 录入方式字段与实际交互深度不匹配。

### 5.6 报表模块

- 可生成周报。
- 可生成月报。
- 可下载最新文件。
- 可查看任务元数据。

当前风险：

- 缺少历史报表列表。
- 缺少处理中、失败、超时的状态反馈。
- 若对象存储未接入，生产环境文件管理能力不足。

### 5.7 社区模块

- 可发帖。
- 可评论。
- 可举报。
- 评论可随帖子返回。

当前风险：

- 缺少分页与排序。
- 缺少用户个人内容管理。
- 缺少内容审核反馈。

## 6. 真实用户最容易感知的问题

以下问题不是代码层面的“是否能运行”，而是用户一上手就会感知到的体验问题，优先级较高。

### 6.1 登录页信息结构不够统一

问题：当前页面同时出现账号、邮箱、手机号三个字段，而登录逻辑实际只走账号字段。  
影响：增加首次使用理解成本，降低注册转化。  
建议：登录页改为“账号 + 密码”，注册弹层或注册页再收集邮箱、手机号。

### 6.2 菜谱模块缺少决策辅助能力

问题：当前菜谱页只能看列表，不能进一步查看详情、营养、收藏、筛选。  
影响：用户浏览后无法立即决定“今天吃什么”。  
建议：补齐搜索、筛选、菜谱详情抽屉和收藏入口。

### 6.3 快速记录流程仍偏后台式填写

问题：当前记录页更像管理表单，不像高频使用工具。  
影响：日常打卡成本偏高，容易影响连续留存。  
建议：增加“今天早餐/午餐/晚餐”快捷入口、常吃菜谱、复制上一餐、拍照/扫码预留入口。

### 6.4 报表体验偏技术化

问题：展示了任务 ID，但对普通用户价值不高。  
影响：用户不知道自己应关注什么，也不清楚报表生成失败时怎么办。  
建议：把“任务 ID”降级为后台信息，强化“本周总结、已达成天数、建议改进点”等用户能看懂的语言。

### 6.5 社区缺少信任反馈

问题：用户举报后看不到处理结果，评论也没有任何审核状态提示。  
影响：用户难以感知平台是否真的在治理内容。  
建议：增加“已提交审核”“处理中”“已处理”的状态反馈。

## 7. 上线前必须补齐项

以下内容建议作为上线闸门，不建议跳过。

### P0：必须完成

- 统一登录交互，消除多字段歧义。
- 隐藏或实现尚未真正落地的记录方式入口。
- 为关键页面补齐空状态、错误态和加载态。
- 完成前端构建验证和 Docker 联调验证。
- 配置生产环境域名、反向代理、文件下载路径。
- 增加基础表单校验与后端异常提示映射。
- 增加最小化数据备份策略和运维检查清单。

### P1：强烈建议完成

- 菜谱搜索、筛选、收藏前端入口。
- 历史报表列表和报表状态轮询。
- 记录编辑、删除与复制上一餐。
- 账号找回密码或至少提供管理员重置流程。
- 社区分页、排序和“我的内容”页。

### P2：后续迭代

- 验证码登录
- 扫码录入
- 对象存储接入
- 推荐反馈闭环
- 运营后台
- 消息通知与提醒

## 8. 建议的验收执行顺序

为了提高验收效率，建议按以下顺序执行：

1. 账号注册与登录
2. 完善个人资料
3. 查看首页推荐与推荐解释
4. 浏览菜谱
5. 新增饮食记录并查看统计
6. 生成周报和月报
7. 发帖、评论、举报
8. 退出登录并重新登录

这样可以完整验证用户从进入系统到形成持续使用闭环的全过程。

## 9. 上线前人工验收样例

### 样例一：减脂用户

- 用户资料：女，身高 165cm，体重 62kg，目标体重 55kg，中等活动。
- 验收点：系统是否能返回合理热量目标，推荐结果是否偏低脂高蛋白，周报是否体现阶段变化。

### 样例二：控糖用户

- 用户资料：男，身高 172cm，体重 75kg，有糖尿病约束。
- 验收点：首页提示语是否切换为控糖方向，推荐理由是否体现少糖或低碳水倾向。

### 样例三：普通记录型用户

- 用户特征：无强健康限制，但希望每天快速记录饮食。
- 验收点：记录流程是否足够短，统计是否易懂，报表是否有复盘价值。

## 10. 结论

从用户视角看，当前系统已经具备较完整的业务闭环，能够支撑毕业设计演示、阶段性验收和继续迭代开发。系统最强的部分在于：核心链路完整、推荐具备解释性、记录和报表形成闭环、工程文档较完整。

但如果以“成熟可上线产品”为标准，当前仍处于“可用 MVP 向产品化过渡”的阶段。最需要优先解决的不是再加很多新功能，而是把已有关键路径做顺手、做清楚、做稳定，尤其是登录、记录、菜谱决策和报表反馈这四个环节。

建议后续所有迭代优先围绕本文件中的 P0 和 P1 项推进。
