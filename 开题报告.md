开题报告

一、课题背景

    随着国民生活水平的不断提高和健康中国战略的深入推进，公众的饮食观念正从"吃得饱"向"吃得好""吃得健康"转变。科学合理的膳食管理已成为慢性病预防、体重控制和整体健康促进的关键环节。《中国居民营养与慢性病状况报告（2020年）》指出，我国居民膳食结构仍存在高盐、高脂、高糖以及营养素摄入不均衡等突出问题，超重和肥胖率持续上升，因饮食不当导致的慢性疾病负担日益加重。由此可见，帮助用户实现科学化、个性化的饮食管理具有重要的现实意义和社会价值。

    然而，当前市面上的饮食健康管理类应用普遍存在以下不足：一是功能模式较为单一，多数应用仅提供简单的热量查询或菜谱浏览，缺乏基于用户个体特征的智能推荐能力；二是营养分析不够深入，多停留在宏量营养素（碳水、蛋白质、脂肪）的粗略估算层面，对微量营养素（维生素、矿物质等）的分析和膳食均衡度评估关注不足；三是用户粘性较差，缺少饮食记录追踪、健康目标管理、社区互动等持续性功能，难以帮助用户形成长期健康的饮食习惯。此外，现有系统的推荐算法大多采用简单的规则匹配或关键词搜索，未能充分利用用户的历史行为数据和偏好特征进行个性化推荐，推荐精准度和用户体验有待提升。

    基于上述背景，本课题拟设计并实现一个"基于Python的智能菜谱推荐与营养健康管理系统"。该系统以用户健康档案为基础，综合运用协同过滤算法和内容推荐算法，为用户提供个性化的菜谱推荐服务；同时整合营养成分分析、饮食记录追踪、健康目标管理、数据可视化等功能，构建一站式的营养健康管理平台。通过该系统，用户不仅能够便捷地获取符合自身需求的菜谱推荐，还能实时掌握营养摄入状况，科学制定和追踪健康目标，从而养成良好的饮食习惯，提升整体健康水平。本课题的研究对于推动智能化健康管理技术在日常饮食领域的应用具有积极的探索意义。

二、国内外研究现状

（一）国外研究现状

    国外在智能菜谱推荐与营养健康管理领域的研究起步较早，研究体系较为成熟，主要围绕推荐算法、营养建模和健康管理平台三个方向展开。在推荐算法方面，协同过滤算法（Collaborative Filtering）和基于内容的推荐算法（Content-based Filtering）是食物推荐领域最为经典的两类方法。协同过滤通过挖掘用户群体的行为相似性来预测个体偏好，而基于内容的推荐则通过分析食物的属性特征（如食材组成、口味、营养成分等）与用户偏好进行匹配。近年来，随着深度学习技术的发展，研究者开始将深度神经网络引入食物推荐领域，利用卷积神经网络（CNN）处理菜谱图像特征、利用循环神经网络（RNN）建模用户的饮食序列行为，在推荐精准度上较传统方法取得了显著提升。此外，知识图谱技术也逐步被应用于食物推荐，通过构建食材、营养素、疾病、用户之间的关联知识网络，实现更具可解释性的健康化推荐。

    在营养管理与膳食规划方面，国外研究注重将推荐系统与营养科学深度整合。一类研究聚焦于"营养约束优化"，即在用户设定的热量、宏量营养素（蛋白质、脂肪、碳水化合物）及微量营养素约束范围内，利用线性规划或启发式算法自动生成最优菜谱组合，有效解决了推荐结果与健康需求脱节的问题。另一类研究关注"健康感知推荐"，将用户的健康状况（如BMI、血糖水平、过敏史等）作为约束条件或权重因子融入推荐模型，使推荐结果在满足口味偏好的同时兼顾健康需求。在商业实践层面，国外已涌现出多个成熟的营养管理平台。MyFitnessPal作为全球用户量最大的饮食追踪应用之一，拥有超过1400万种食物的营养数据库，支持用户记录每日饮食并自动计算营养摄入。Yummly则将个性化推荐引擎与菜谱平台深度结合，基于用户的口味偏好、饮食限制和营养目标进行智能菜谱推荐。Cronometer以专业级的微量营养素追踪功能著称，被广泛应用于临床营养管理。这些平台在数据驱动的推荐模式、营养数据库建设和用户体验设计方面积累了丰富的工程经验，为本课题的系统设计提供了重要参考。

（二）国内研究现状

    国内在智能菜谱推荐与营养健康管理方面的研究近年来发展迅速，技术应用不断深化。在推荐算法研究方面，陈向峰等（2025）探讨了大语言模型在智能菜谱推荐中的应用，通过预训练语言模型理解用户自然语言描述的饮食需求，生成更符合用户意图的菜谱推荐结果，代表了该领域的前沿方向。在系统开发层面，国内研究者多采用前后端分离的架构模式进行系统设计。后端开发以Python的Django框架和Java的Spring Boot框架为主流，前端则普遍采用Vue.js框架结合Element Plus等组件库构建用户界面。数据库方面，MySQL和PostgreSQL被广泛使用，Redis常被引入作为缓存层以提升系统响应性能。

    在营养管理实践方面，国内涌现了薄荷健康、Keep饮食等多款营养管理应用，实现了食物热量查询、饮食记录、营养报告等基础功能。然而，这些应用在推荐算法的精准度、营养分析的全面性以及功能模块的集成度方面仍有提升空间。多数国内系统的推荐功能局限于简单的标签匹配和热门排序，缺乏基于用户行为数据的深层次个性化推荐能力；营养分析多依赖于静态的食物营养成分表，未能结合用户个体的健康状况和营养需求进行动态评估和建议。此外，社区互动和健康目标追踪等功能在多数系统中未得到充分整合，导致用户留存率和使用活跃度不高。

    综上所述，尽管国内外在智能菜谱推荐与营养健康管理领域已取得一定研究成果，但在推荐算法的个性化程度、营养分析的全面性、功能模块的一体化集成以及用户长期使用体验等方面仍存在不足。本课题将在已有研究基础上，综合运用协同过滤算法与内容推荐算法，构建一个集智能菜谱推荐、营养成分分析、饮食记录追踪、健康目标管理、社区互动与数据可视化于一体的综合性营养健康管理系统，力求在推荐精准度和功能集成度方面实现改进与提升。

三、研究的主要问题

（一）系统功能设计

    本系统围绕真实饮食管理场景，已经形成“建档—推荐—记录—目标—报表—社区—提醒—AI 助手—付费订阅—后台运营”的完整业务闭环，当前研究重点主要体现在以下几个方面：

    1. 用户画像与健康约束的一体化管理

    系统需要同时支撑注册登录、JWT 鉴权、密码重置、安全问题验证、头像上传、账号注销等账号能力，以及用户档案、慢病状态、过敏与忌口信息的长期维护。这些信息既是推荐、营养分析、报表和 AI 助手的输入基础，也是后续个性化约束和权限控制的统一来源，因此必须保证模型结构清晰、字段约束稳定、更新链路一致。

    2. 混合推荐与冷启动处理

    当前推荐模块并非单一规则排序，而是协同过滤、基于内容推荐和规则打分三路混合。重点问题在于：如何把用户行为、收藏和餐食记录组织成稀疏交互矩阵；如何把菜谱标签和用户偏好编码成统一向量；以及当新用户缺少历史行为时，如何通过内容推荐与规则打分平稳接管冷启动场景，保证推荐链路不中断。

    3. 个性化营养评估与阶段复盘

    系统需要根据年龄、性别、活动水平、目标体重和健康状况，输出每日热量与多项营养素目标，并对近期饮食记录进行多维聚合，生成营养充足率、膳食均衡评分、趋势图和阶段性 PDF 报告。如何在统一模型下贯通首页、记录页、报表页和 AI 助手，是系统设计中的关键问题。

    4. 多模块业务一致性与后台治理

    当前系统已经扩展出社区审核、敏感词规则、公告中心、站内提醒删除、帖子归档与彻底删除、后台操作日志等治理能力。如何保证不同角色、不同入口下的数据状态一致，并避免“后台删除但提醒未联动”“前端已更新但服务端规则未生效”等不一致问题，是当前系统设计的现实重点。

    5. 低配置服务器下的工程化落地

    生产环境部署在 2 核 2GiB 的单机云服务器上，系统必须控制构建和运行资源占用。前端采用本地构建后同步 dist 到服务器，后端通过 Docker Compose、Gunicorn、Celery 与 Redis 协作，既要保证推荐预计算、订阅到期检查、报表生成等异步任务稳定运行，又要尽量减少部署阶段对服务器资源的冲击。

（二）本课题的难点

    1. 稀疏行为数据下的推荐准确性

    推荐模块使用用户行为、收藏与饮食记录构造稀疏交互矩阵，并基于余弦相似度计算相似用户。由于多数用户行为分布离散、互动量有限，如何在稀疏数据下保持协同过滤有效，同时避免新用户完全失去推荐能力，是推荐链路中的核心难点。

    2. 个体化营养指标的统一建模

    用户在年龄、性别、活动水平、目标体重、慢病状态等方面差异很大，营养分析服务需要在统一数据模型下输出热量、蛋白质、脂肪、碳水、膳食纤维、钠、钙、铁和维生素 C 等目标值，并把这些指标贯通到首页、记录页、报表页和 AI 助手中，保证解释口径一致。

    3. 复杂业务链路的一致性维护

    系统当前涉及推荐、营养分析、社区互动、后台审核、公告推送、站内提醒、AI 助手、支付订阅等多条业务链路。如何保证各模块之间的数据流转正确、权限边界清晰，并避免后台治理动作与前台用户态不同步，是当前实现中的重要难点。

    4. 异步任务、缓存与低资源部署的平衡

    推荐预计算、订阅到期检查、报表生成等任务都依赖 Celery / Celery Beat 与 Redis 协作完成；前端资源又要求本地构建后上传，尽量减少服务器负担。如何在有限资源条件下平衡同步请求、异步任务、缓存刷新与部署流程，是系统工程实现中的现实挑战。

四、拟采取的技术和方法

（一）系统架构设计

    系统采用严格的前后端分离 B/S 架构。前端为 Vue 3 单页面应用，浏览器请求由 Nginx 统一入口处理；`/api/v1/` 路径转发至 Django REST Framework 提供的 RESTful API；静态页面和前端路由由 SPA 自行接管。生产部署采用 Docker Compose 编排 backend、worker、beat、frontend、db、redis 等多个服务，前端采取本地构建生成 dist 后上传服务器的方式，降低低配云服务器的构建压力。

（二）后端技术

    后端实际采用 Python 3.11+、Django 5.2 与 Django REST Framework 构建。PostgreSQL 作为主数据库（开发环境支持 SQLite 自动回退），Redis 同时承担 Celery Broker、结果后端与部分缓存职能。系统使用 djangorestframework-simplejwt 实现 JWT 鉴权，Pillow 处理图片上传与压缩，ReportLab 生成 PDF 报告，Celery + Celery Beat 承担推荐预计算、订阅到期检查等异步任务。除业务接口外，系统还实现了管理员操作日志、社区敏感词规则、站内提醒、后台公告中心、社区审核与规则管理等后台治理能力。

（三）前端技术

    前端采用 Vue 3 + TypeScript + Vite 工具链构建，Vue Router 管理页面路由，Pinia 负责认证态与全局状态，Axios 用于 HTTP 通信。界面组件主要基于 Element Plus，图表展示采用 ECharts。页面已覆盖首页工作台、饮食记录、菜谱库、收藏、目标、报表、社区、公开用户主页、个人档案、AI 助手、定价与支付结果，以及后台总览、公告中心、用户管理、社区审核、敏感词管理、运营复核、菜谱管理和操作日志等后台页面。

（四）算法与模型方法

    1. 混合推荐算法（hybrid_v1）

    推荐引擎采用三路混合方案：协同过滤（40%）+ 基于内容推荐（30%）+ 规则打分（30%）。协同过滤部分使用 UserBehavior、UserFavoriteRecipe 与 MealRecordItem 构建稀疏用户-菜谱交互矩阵，通过 Python 标准库 `math` 手写余弦相似度，计算用户间相似性并选取 Top-20 相似用户完成加权聚合；基于内容推荐部分将菜谱编码为固定维度的标签特征向量，并将用户的 diet_type、meal_preference 与历史收藏特征融合为偏好向量，再计算余弦相似度；规则打分部分综合营养指标、烹饪时长、饮食类型和约束条件直接赋分。对于冷启动用户，当协同过滤无法得到有效分数时，其权重自动转移至内容推荐与规则打分，保证推荐链路不中断。

    2. 个性化营养评估模型

    营养评估服务依据用户年龄、性别、活动水平等信息，按 DRIs 标准分档计算每日热量、蛋白质、脂肪、碳水化合物、膳食纤维、钠、钙、铁和维生素 C 的推荐摄入量；同时根据用户身高和体重实时计算 BMI，并结合糖尿病、高血压、高血脂等健康状态输出个性化饮食提示。系统对近期饮食记录进行聚合，计算各营养素摄入充足率与综合膳食均衡评分，为首页提示、报表生成和 AI 助手回答提供统一分析基础。

    3. 报表生成与数据可视化方法

    报表服务通过 Django ORM 与业务服务层对指定时间段内的餐食记录进行多维聚合，输出 14 天每日营养趋势、餐型分布、宏量营养素分布、星期规律、30 天活跃热力图和餐次完成率等结构化数据集及洞察文本。前端使用 ECharts 进行交互式图表展示，PDF 报告由后端使用 ReportLab 程序化生成 A4 多页文档；系统实际未使用 Matplotlib 生成报表图表。

    4. AI 助手与多模态识别方法

    系统额外实现了 AI 助手模块。基于阿里云通义千问（DashScope，OpenAI 兼容接口）提供 SSE 流式对话，系统提示词会动态融入用户健康档案、活跃目标、近 7 日饮食记录、收藏菜谱及后台页面上下文。食物图片识别功能使用视觉模型解析上传图片，并要求模型返回结构化 JSON，对食材、份量、步骤和营养估算进行统一抽取与清洗。

    开发语言：Python 3.11+、TypeScript

    后端框架：Django 5.2、Django REST Framework

    前端框架：Vue 3、Vite、Vue Router、Pinia、Axios、Element Plus、ECharts

    开发工具：Visual Studio Code、Git、Postman、Docker Desktop
